# Spec Delta

## Added

### Задача 1 — strip `ПРЕДСТАВЛЕНИЕ()` в запросе

- Перед отправкой в 1С шлюз вычищает вызовы платформенных функций получения строкового представления: `ПРЕДСТАВЛЕНИЕ(X)`, `ПРЕДСТАВЛЕНИЕССЫЛКИ(X)`, `PRESENTATION(X)`, `REFPRESENTATION(X)`. На месте вызова остаётся сам аргумент `X` — 1С возвращает ссылку, а не строку.
- Очистка применяется и к MCP-пути (`toolQuery`), и к UI-пути (`HandleQuery`).
- Строковые литералы `"..."` и комментарии `//` от очистки защищены.
- Очистка идёт до нормализации алиасов (`normalizeQueryAliases`).

### Задача 3 — type-aware маскирование

- При каждом MCP-запросе к 1С шлюз передаёт аргумент `IncludeSchema: true` и ожидает JSON-ответ `{version, columns: [{name, types, truncated?, types_total?}], rows}`. На TSV-ответ (старый бэкенд) — fallback без схемы.
- Схема типов колонок сохраняется в сессии (`ColumnTypes`, `ColumnTruncated`).
- Маскирование применяет **политику по типу колонки** как первый слой решения — до whitelist'а по именам:
  - По умолчанию **plain** для типов: `Перечисление.*`, `ПланСчетов.*`, `ПланВидовХарактеристик.*`, `Справочник.Валюты`, `Справочник.СтавкиНДС`, `Справочник.СтраныМира`, `Справочник.БанкиРФ`, примитивы `Число/Дата/Булево/УникальныйИдентификатор`.
  - По умолчанию **mask** для: `Справочник.*` (кроме классификаторов), `Документ.*`, `Задача.*`, `БизнесПроцесс.*`, `Строка`.
  - Колонка с `truncated == true` → всегда **mask** (среди скрытых типов могут быть опасные).
  - Составной тип: **один опасный тип → вся колонка masked**.
- Политика персистится в credential store (`PersistentTypePolicy`), сливается с хардкод-дефолтами.
- В UI появляется раздел **«Политика типов»** — таблица `Тип/Префикс | Политика | Комментарий` с возможностью редактирования.
- В шапке/тултипе каждой колонки таблицы результатов отображается её тип и, если применимо, метка `truncated (N типов)`.

### Сценарии

**Сценарий A — агент запросил представление документа**:
- Агент: `ВЫБРАТЬ ПРЕДСТАВЛЕНИЕ(Регистратор) КАК РегПредст ИЗ РегистрНакопления.ДвиженияТоваров`.
- Шлюз: strip + normalize → `ВЫБРАТЬ Регистратор КАК Регистратор ИЗ ...`.
- 1С возвращает колонку `Регистратор` с типом `Документ.*` (составной, >10 типов) + `truncated: true`.
- Шлюз маскирует колонку (политика). Пользователя не беспокоим.
- **Проверка**: `go test ./... -run TestStripPresentation` + smoke на реальной базе.

**Сценарий B — запрос с полями-перечислениями и счетами**:
- Агент: `ВЫБРАТЬ Период, ВидДвижения, СчетДт, СчетКт, Сумма ИЗ РегистрБухгалтерии.Хозрасчетный`.
- 1С возвращает типы: `Дата`, `Перечисление.ВидыДвижения`, `ПланСчетов.Хозрасчетный`, `ПланСчетов.Хозрасчетный`, `Число`.
- Шлюз: все 5 колонок автоматически **plain** по политике. Ни одного `gateway_suggest_fields`.
- **Проверка**: unit-тест `TypePolicy.Decide`, интеграционный тест с моковым JSON-ответом.

**Сценарий C — составной тип с обрезанием**:
- Запрос возвращает колонку `Регистратор` с `types: [10 документов]`, `truncated: true`, `types_total: 47`.
- Шлюз: **mask** по политике обрезания.
- В UI колонка помечена «Составной тип (47 вариантов)».
- **Проверка**: unit-тест `TypePolicy.Decide({truncated: true, ...})` → mask.

**Сценарий D — fallback на TSV**:
- Старый 1С-бэкенд без `IncludeSchema` возвращает TSV.
- Шлюз парсит TSV, `ColumnTypes` пустые, политика возвращает `unknown` на все колонки.
- Работает старая логика: whitelist по именам, skipNumeric, NER. В UI — предупреждение: «Схема типов недоступна — обновите MCP-сервер 1С для автоматической классификации колонок».
- **Проверка**: интеграционный тест с TSV-ответом.

## Updated

- Описание в `mcp_server.go`: убрать акцент на «попроси suggest для перечислений». Добавить: «Поля-перечисления, планы счетов, числа, даты автоматически открыты по политике типов».

### Инструмент `gateway_suggest_fields` (MCP)

- Область применения сужается: остаётся для ссылочных типов, не покрытых политикой. Описание обновить: «Используй, если колонка со справочным типом, не входящим в белый список политики типов, — и тебе нужно её содержимое в открытом виде».

### Acceptance

- На исходном 40-полевом запросе пользователя (Код, ВидСубконтоПредставление, Период, ПРВидДвижения, СчетДт/Кт, СтатьяРасходов, МоментВремени, ТипРасходов, ВариантРаспределения*, Регистратор и т.д.) после реализации: **0 вызовов** `gateway_suggest_fields` для полей-перечислений/дат/планов счетов/кодов/булевых.
- Маскируются только реально чувствительные: Регистратор (составной документ), Контрагент (если попадёт), СтатьяРасходов (Справочник).
- `go test ./...` — зелёный: новые тесты `query_normalize_test.go`, `type_policy_test.go`, intergation-тесты в `privacy_test.go`, все существующие.
- Smoke-прогон на реальной базе 1С пользователя: один-два запроса руками + через агента.
- UI: раздел «Политика типов» виден, редактируется, сохраняется, загружается между перезапусками.
