# Proposal

## Change ID
`2026-04-14-type-aware-masking`

## Контекст

- **Идея**: перевести маскирование результатов запросов с whitelist'а по именам полей на классификацию по типу колонки из схемы метаданных 1С.
- **Для кого**: пользователь-владелец шлюза (разработчик), работающий с агентом, которому нужно давать доступ к данным 1С без ручного разрешения десятков полей на каждый запрос.
- **Проблема**: агент на один 40-полевой запрос присылает массовый `gateway_suggest_fields` (пример: «Код, ВидСубконтоПредставление, ПРЕДСТАВЛЕНИЕРегистратор, Период, ПРВидДвижения, ВариантРаспределения*, СчетДт, …»). Все эти поля очевидно безопасны (перечисления, даты, счета, коды), но шлюз не знает их типа — работает только по именам, поэтому вынуждает пользователя одобрять каждое.

## Что меняем

1. **Задача 1 — strip `ПРЕДСТАВЛЕНИЕ()` в Go-шлюзе** (`query_normalize.go`). Шлюз перед отправкой в 1С вычищает вызовы `ПРЕДСТАВЛЕНИЕ/ПРЕДСТАВЛЕНИЕССЫЛКИ/PRESENTATION/REFPRESENTATION`, подставляя на их место аргумент. Платформа возвращает саму ссылку, тип сохраняется, шлюз получает возможность классифицировать колонку.

2. **Задача 3 — type-aware маскирование в шлюзе**. Шлюз при каждом MCP-запросе передаёт в 1С `IncludeSchema: true`, парсит JSON-ответ со схемой колонок, хранит карту `column → types[]` в сессии и применяет политику по типам вместо/дополнительно к whitelist'у имён. В UI появляется раздел «Политика типов».

Задача 2 (параметр `IncludeSchema` и JSON-формат со схемой на стороне 1С MCP) уже выполнена — см. `MCP from 1C/src/DataProcessors/mcp_ИнструментЗапросыКБазе/Ext/ManagerModule.bsl`.

## Вне границ

- Классификация ещё **до** выполнения запроса через `СхемаЗапроса` на стороне 1С (отдельная задача — можно будет добавить, чтобы блокировать опасные запросы заранее).
- Автообучение «безопасных» справочников на основе статистики запросов.
- Переработка NER-эвристик в `ner.go`.
- Выкидывание whitelist'а по именам — он остаётся как дополнительный слой, просто перестаёт быть основным.
- Изменения на стороне `MCP from 1C` сверх уже сделанного.

## Критерии готовности

- Запросы агента с `ПРЕДСТАВЛЕНИЕ(X)` автоматически преобразуются — в 1С уходит `X`.
- Колонки с типом `Перечисление.*`, `ПланСчетов.*`, `Число`, `Дата`, `Булево`, `УникальныйИдентификатор` по умолчанию plain — без запроса к пользователю.
- Колонки с типом `Справочник.*`/`Документ.*`/`Задача.*`/`БизнесПроцесс.*`/`Строка` по умолчанию masked — кроме типов, явно добавленных пользователем в whitelist политики.
- Колонки с `truncated: true` (составной тип >10 вариантов) — masked по умолчанию.
- В UI есть раздел «Политика типов» с редактируемой таблицей и предзаполненными разумными дефолтами.
- На примере исходного 40-полевого запроса пользователя — 0 запросов `gateway_suggest_fields` для полей-перечислений, дат, счетов, кодов.
- `go test ./...` зелёный: новые тесты + все существующие.
- Smoke-прогон на реальной базе пользователя.
