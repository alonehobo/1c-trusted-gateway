# Design

## Контекст

- **Компоненты**: Go-шлюз (`trusted-gateway-go`) в части нормализации запросов, HTTP-клиента к 1С MCP, парсинга ответа, слоя sanitize, сессии и UI.
- **Внешние интеграции**: 1С MCP-сервер (`MCP from 1C`) — по HTTP, инструмент `query`. На стороне 1С уже реализован параметр `IncludeSchema: true` и JSON-ответ `{version, columns:[{name,types,truncated?,types_total?}], rows}`. Значения ячеек — строки (как в TSV), но теперь известен тип каждой колонки.
- **Затронутые файлы**: `query_normalize.go`, `query_normalize_test.go`, `mcp_client.go`, `service.go`, `privacy.go`, `web.go`, `http_handlers.go`, `ui.html`, `credential_store.go`, новый `type_policy.go` + `type_policy_test.go`.

## Подход

### Задача 1 — strip `ПРЕДСТАВЛЕНИЕ()`

Новая функция `stripPresentationCalls(query string) string` в `query_normalize.go`. Идея: токенизируем посимвольно, как существующие функции этого файла; встречая идентификатор из банлиста, сопровождающийся `(`, находим парную `)` и заменяем всё `ИМЯ(...)` на содержимое скобок.

Банлист (case-insensitive): `ПРЕДСТАВЛЕНИЕ`, `ПРЕДСТАВЛЕНИЕССЫЛКИ`, `PRESENTATION`, `REFPRESENTATION`.

Ключевые требования к реализации:
- **Уважать вложенные скобки** — аргумент может быть `ВЫБОР КОГДА А ТОГДА Б КОНЕЦ`, вложенным вызовом, арифметикой.
- **Пропускать строковые литералы** `"..."` с учётом 1С-экранирования `""`.
- **Пропускать комментарии** `//` до конца строки.
- **Вложенные случаи** — несколько проходов до стабилизации, с жёстким лимитом 8 итераций на защиту от зацикливания.
- **Не трогать поле/алиас с именем `Представление` без следующей `(`** — это обычный идентификатор.

Порядок вызова в MCP- и UI-путях:
1. `stripPresentationCalls(queryText)` (новое)
2. `normalizeQueryAliases(queryText)` (существующее)

Это даёт правильный финальный алиас: `ПРЕДСТАВЛЕНИЕ(Регистратор) КАК РегПредст` → после strip → `Регистратор КАК РегПредст` → после normalize → `Регистратор КАК Регистратор`.

Сторонний эффект (осознанный): если агент использует `ПРЕДСТАВЛЕНИЕ()` в `ГДЕ` для сравнения со строкой, после strip запрос упадёт в 1С (ссылка ≠ строка). Это ожидаемо — агент получит платформенную ошибку и переформулирует. Хитрить со «strip только в SELECT» не стоит, парсер станет хрупким.

### Задача 3 — type-aware маскирование

**Слой 1 — клиент 1С MCP (`mcp_client.go`)**. При вызове инструмента `query` в аргументах добавляется `IncludeSchema: true`. Остальные инструменты не трогаем.

**Слой 2 — парсер ответа (`service.go`)**. Перед текущим `parseTabularText` добавляется попытка разобрать JSON формата `{version, columns, rows}`. Если удалось — извлекаем `columns[]` (имя + типы + флаги обрезания) и `rows[]`; иначе — fallback на TSV-парсинг (обратная совместимость со старыми 1С-бэкендами).

**Слой 3 — хранение схемы (`web.go`)**. В `TrustedSession` добавляется:
```go
ColumnTypes     map[string][]string    // column name (normalized) → type full names
ColumnTruncated map[string]int         // column name → total types count (если обрезано)
```

Заполняется при успешном приёме запроса, очищается при смене сессии.

**Слой 4 — политика типов (`type_policy.go`)**. Новый модуль:
```go
type TypePolicy struct {
    plainTypes   map[string]bool      // точные соответствия: "Перечисление.ВидыДвижения", "Справочник.Валюты"
    plainPrefixes []string            // семейства: "Перечисление.", "ПланСчетов.", "ПланВидовХарактеристик."
    forcedMask   map[string]bool      // типы, всегда маскируемые, перекрывают plain
}

func (tp *TypePolicy) Decide(types []string, truncated bool) Decision
// Decision: { policy: "plain" | "mask" | "unknown", reason: string }
```

Правила решения:
1. **Если `truncated == true`** → всегда `mask` (среди скрытых могут быть опасные типы).
2. **Если пустой `types[]`** → `mask` (неопределённый тип).
3. **Иначе**: колонка `plain` **только если все** типы из её списка проходят политику `plain` и ни один не в `forcedMask`. Один опасный тип в составном — маскируется вся колонка.
4. Примитивы `Число/Дата/Булево/УникальныйИдентификатор` — `plain` по умолчанию (вшиты в политику).
5. Примитив `Строка` — `mask` по умолчанию (свободный текст потенциально содержит ПДн).

Дефолты политики (можно выключить в UI):
- plain: `Перечисление.*`, `ПланСчетов.*`, `ПланВидовХарактеристик.*`, `Справочник.Валюты`, `Справочник.СтавкиНДС`, `Справочник.СтраныМира`, `Справочник.БанкиРФ`.
- mask: всё остальное.

**Слой 5 — интеграция в `DataSanitizer` (`privacy.go`)**. Порядок принятия решения в `maskValue` обновляется:
1. Nested map/slice → рекурсия (как сейчас).
2. `bool` → plain (как сейчас).
3. `forceMask[name]` → mask (как сейчас).
4. **Новый шаг**: `TypePolicy.Decide(columnTypes[name], truncated[name])`:
   - `plain` → возвращаем значение как есть.
   - `mask` → падаем в общую маскировку.
   - `unknown` → продолжаем к следующим шагам (ведём себя как сейчас).
5. `allowPlain[name]` → plain (существующее — ручной whitelist по имени).
6. `matchesAllowKeyword` → plain (существующее).
7. `skipNumeric + isRealNumber` → plain (существующее).
8. Boolean-like string → plain (существующее).
9. Empty → pass.
10. Mask.

Логика: политика типов — **дополнительный, более приоритетный, слой**. Whitelist по именам остаётся как escape hatch для случаев, когда у колонки неизвестен тип (TSV-fallback или schema отсутствует).

**Слой 6 — персистентная политика (`credential_store.go`, `web.go`)**. Новое поле `PersistentTypePolicy string` (JSON в credential store). Формат:
```json
{
  "plain_types": ["Справочник.Валюты"],
  "plain_prefixes": ["Перечисление.", "ПланСчетов."],
  "forced_mask_types": []
}
```
На старте загружается, сливается с hardcoded-дефолтами.

**Слой 7 — UI (`ui.html`)**. Новый раздел «Политика типов»:
- Таблица: `Тип/Префикс | Политика (plain/mask) | Комментарий | ✕`.
- Кнопка «Добавить правило».
- Подсказка: «Префикс должен заканчиваться на точку (`Перечисление.`). Точное имя — без точки в конце (`Справочник.Валюты`)».
- Дефолтные записи отмечены серым, можно «отключить» (помещается в forcedMask… нет, лучше просто `enabled: false` в структуре).

В таблице результатов для каждой колонки в шапке/тултипе отображается тип (`ПеречислениеСсылка.ВидыДвижения`) и флаг `truncated (N типов)` если есть. Это даёт пользователю контекст для ручного редактирования политики.

## Основные объекты

- **Компоненты**: Go-бинарь `trusted-gateway-go`.
- **Модули/пакеты**: новый `type_policy.go` с тестами; правки в `query_normalize.go`, `mcp_client.go`, `service.go`, `privacy.go`, `web.go`, `http_handlers.go`, `credential_store.go`, `ui.html`.
- **Данные**:
  - В сессии: `ColumnTypes map[string][]string`, `ColumnTruncated map[string]int`.
  - В app: `PersistentTypePolicy string` (JSON в credential store).
- **API и интеграции**: в исходящих MCP-вызовах `tools/call → query` добавляется аргумент `IncludeSchema: true`. В ответе теперь парсится JSON `{version, columns, rows}` с fallback на TSV.
- **UI**: новая вкладка/аккордеон «Политика типов» в `ui.html`; индикатор типа в шапке таблицы результатов.

## Риски и ограничения

- **Обратная совместимость с 1C MCP**: если бэкенд ещё старый (без `IncludeSchema`) — ответ придёт TSV. Нужен fallback: схема пустая, классификация уходит в «unknown», работает как раньше по whitelist имён. Пользователь должен получить подсказку в UI: «Схема типов недоступна — обновите MCP-сервер на стороне 1С».
- **Составные типы с обрезанием**: политика жёсткая (mask). Можно будет добавить доп. кнопку «запросить полный список типов у 1С» — не в рамках этого change.
- **Агрегатные функции (СУММА, МАКСИМУМ)**: у платформы тип результата вычисляется, `Колонка.ТипЗначения.Типы()` корректен. Проверить вручную на smoke.
- **ВЫБОР-выражения с разнотипными ветками**: тип колонки — объединение всех веток. Если одна из веток — Справочник, вся колонка masked. Это корректное поведение.
- **Strings в результатах**: по умолчанию masked. Это потеряет удобство для колонок типа `Код` справочника (они Строка). Mitigation: добавить в UI-дефолты правило «`Код` у справочников-классификаторов — plain» через allowKeywords или через white-список типов `Справочник.*` с отдельным плагинным полем. Альтернатива: при ответе 1С включать `is_code_column: true` (из `Колонка.Имя == "Код"` в результате). Пока держим просто: Строка → mask, пусть пользователь добавит исключения в существующий name-whitelist.
- **Замена ПРЕДСТАВЛЕНИЕ() в WHERE/УПОРЯДОЧИТЬ ПО**: запрос может не выполниться в 1С. Ожидаемо, обрабатывается на уровне 1С-ошибки, агент переформулирует.
- **Производительность**: парсинг JSON вместо TSV — сопоставимая стоимость, но JSON больше. Для типичного MCP-запроса (десятки—сотни строк) несущественно.

## Стратегия проверки

**Unit-тесты**:
- `query_normalize_test.go`:
  - все 4 варианта strip на базовом `ВЫБРАТЬ ... КАК`
  - вложенный `ПРЕДСТАВЛЕНИЕ(ПРЕДСТАВЛЕНИЕ(X))`
  - сложный аргумент с `ВЫБОР` / арифметикой
  - поле с именем `Представление` без `(` не трогается
  - `"ПРЕДСТАВЛЕНИЕ(X)"` в строковом литерале не трогается
  - `// ПРЕДСТАВЛЕНИЕ(X)` в комментарии не трогается
  - strip + normalize вместе дают ожидаемый финальный алиас
- `type_policy_test.go`:
  - таблица `{types, truncated} → expected Decision` (15-20 кейсов, все категории)
  - приоритет forcedMask над plain
  - составной тип с одним опасным → mask
  - пустой types/truncated=true → mask
- `privacy_test.go` (если ещё нет, создать): интеграция политики с `SanitizeRows` на фикстурном JSON-ответе.

**Smoke на реальной базе**:
- Запросить `ВЫБРАТЬ Период, ВидДвижения, СчетДт, СчетКт, Сумма, Контрагент, ПРЕДСТАВЛЕНИЕ(Регистратор) КАК Рег ИЗ РегистрБухгалтерии.Хозрасчетный`.
- Убедиться: в 1С приходит без `ПРЕДСТАВЛЕНИЕ`; результат: первые 5 колонок — plain, Контрагент — masked, Рег (переименован в Регистратор) — masked с пометкой составного типа.
- Убедиться: ни одного `gateway_suggest_fields` не пришло.

**Regression**:
- Все существующие тесты `query_normalize_test.go` продолжают проходить.
- Существующая UI-функциональность (персистентный whitelist имён, force-mask, skipNumeric) работает как раньше.
